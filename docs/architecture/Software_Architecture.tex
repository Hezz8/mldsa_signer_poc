\chapter{Software Architecture}

\section*{Purpose}
This chapter defines the current Linux-side software structure hosted on the Zynq PS.

\section{Current Implemented Structure}
The repository now contains both a deterministic local development path and a target-facing real-MMIO bring-up path. The public software contract remains unchanged: the daemon accepts a 64-byte digest, sequences the MMIO device abstraction, and returns signature bytes plus status.

\section{Current Components}
\begin{itemize}[leftmargin=*]
  \item \textbf{Service core:} transport-independent signing orchestration over the device abstraction.
  \item \textbf{MMIO device abstraction:} stable software API for digest writes, start, wait, readback, and clear-status behavior.
  \item \textbf{Fake backend:} deterministic local register model used for unit tests, local self-test, and client smoke checks.
  \item \textbf{Real backend:} PoC user-space MMIO backend intended for Linux on Zynq, using a `/dev/mem`-style mapping path and the documented wrapper base address.
  \item \textbf{Probe helper:} safe register-visibility check path that avoids triggering signing unless explicitly requested elsewhere.
  \item \textbf{Optional gRPC transport:} runtime-generated Python gRPC binding path enabled when dependencies are installed.
\end{itemize}

\section{Backend Selection}
Backend selection is configuration-driven. The default remains `fake` so local development continues to work exactly as before. The `real` backend is selected through environment variables or command-line overrides and requires platform-specific MMIO mapping inputs such as the wrapper base address.

\section{Operational Model}
In fake mode, the daemon preserves the existing deterministic stub behavior. In real mode, the daemon uses the same high-level device API but routes reads and writes through a memory-mapped access path intended for Linux on the target board. The probe path reads status-oriented registers only, while explicit self-test can drive a signing sequence when hardware is present.

\section{Safety Boundaries}
The real backend is intentionally practical rather than hardened. It assumes little-endian 32-bit register access, page-aligned mapping behavior, and a correct AXI-Lite base address supplied by the platform design. Local non-target development is unaffected because the fake backend remains available and is still the default.

\section{Current Claim Boundary}
The repository now contains the software infrastructure required for first PS-to-PL bring-up, but it does not yet claim that target-board register access or end-to-end signing has been validated on actual Zynq hardware.
