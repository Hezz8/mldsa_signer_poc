\chapter{Hardware Microarchitecture}

\section*{Purpose}
This chapter describes the current PL-side organization at a level suitable for integration planning and early verification.

\section{Current Block Structure}
\begin{itemize}[leftmargin=*]
  \item \textbf{AXI-Lite wrapper:} register decode, digest staging, command acceptance, status reporting, and signature readback windows.
  \item \textbf{ML-DSA engine adapter:} stable internal boundary that accepts wrapper-level start and digest inputs and returns busy, done, error, signature length, and signature data.
  \item \textbf{Future ML-DSA core attachment point:} reserved seam where ML-DSA-OSH-specific control, buffering, and datapath signals will attach later.
  \item \textbf{Signature buffer exposure:} wrapper-visible byte window populated from adapter output once an operation completes.
\end{itemize}

\section{Current Adapter Modes}
The current adapter supports two deterministic non-cryptographic modes:
\begin{itemize}[leftmargin=*]
  \item \textbf{STUB mode:} reproduces the existing fake-signature behavior used to validate wrapper sequencing and software integration.
  \item \textbf{CORE\_PLACEHOLDER mode:} preserves the same adapter contract while standing in for a future ML-DSA-OSH-facing engine implementation.
\end{itemize}

\section{Clocking and Reset}
The initial implementation assumes a single primary PL clock domain for the wrapper and adapter path. Reset behavior forces the wrapper into an idle, non-busy state with cleared status and an empty signature window.

\section{Dataflow}
Software writes the digest into wrapper-visible registers, asserts `start`, the wrapper launches the engine adapter, and the adapter later reports completion with a signature buffer and signature length. The wrapper then exposes the result through the PS-readable `SIG\_LENGTH` and `SIG\_DATA` windows.

\section{Adapter Rationale}
The adapter prevents the AXI-Lite wrapper from depending directly on ML-DSA-OSH-specific handshake or datapath details. This allows the PS-visible contract to remain stable while the downstream engine implementation evolves from deterministic scaffolding to a real hardware core.

\section{Microarchitecture Constraints}
The microarchitecture shall support repeated signing requests without requiring PL reconfiguration or manual state cleanup between successful transactions. The wrapper contract shall remain stable across the upcoming transition from `CORE\_PLACEHOLDER` to real ML-DSA integration.
