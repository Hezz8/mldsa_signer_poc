\chapter{Preliminary Design Review}

\section*{Purpose}
This chapter captures the current architectural direction for the proof-of-concept appliance.

\section{System Objective}
The system accepts a 64-byte digest from a remote client, forwards it through a Linux-resident gRPC service, sequences a hardware signing request through AXI-Lite, and returns the resulting ML-DSA-87 signature.

\section{Top-Level Partition}
\begin{itemize}[leftmargin=*]
  \item \textbf{Client side:} submits signing requests over Ethernet.
  \item \textbf{PS software:} runs Linux, hosts the gRPC daemon, validates requests, manages MMIO, and reports status.
  \item \textbf{PL logic:} hosts the stable AXI-Lite wrapper, the ML-DSA engine adapter, project-owned ML-DSA-OSH shim logic, and the imported signing datapath.
\end{itemize}

\section{Key Architectural Choices}
\begin{itemize}[leftmargin=*]
  \item Use AXI-Lite for the initial PS and PL boundary to minimize integration complexity.
  \item Keep the external API pre-hash oriented so the service boundary is simple and deterministic.
  \item Keep the wrapper contract independent from downstream engine implementation details.
  \item Preserve a deterministic `STUB` path for regression and first board bring-up.
  \item Add a real MMIO backend in software without removing the fake backend used for local development.
\end{itemize}

\section{Current Implementation Boundary}
The repository now includes a real imported ML-DSA-OSH source snapshot, a project-owned hardware shim behind the adapter, and a software real-MMIO backend intended for target-board bring-up. The gRPC API, software device API, and wrapper-visible register contract remain stable.

\section{What Is Verified vs Provisional}
Local software behavior, deterministic hardware wrapper behavior, and documentation consistency are verified. The software real backend is implemented and locally sanity-tested against file-backed mappings, but actual PS-to-PL communication on Zynq remains to be verified on target hardware. Full local mixed-language simulation of the imported upstream sign path also remains tooling-limited.

\section{Deferred Topics}
Deferred topics include production key management, hardened device-access mechanisms, end-to-end target-board validation, peak throughput optimization, and packaging for field deployment.
